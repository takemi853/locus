# ナレッジベースのインデックス検索（RAGなし）

個人スケールのナレッジベース（50〜500記事）では、ベクトル検索（RAG）を使わず、LLMが構造化インデックスファイルを直接読んで関連記事を選択するアプローチが優れている。Andrej Karpathy の知見に基づく設計判断。

## Key Points

- `knowledge/index.md` がマスターカタログ — 全記事の1行サマリーを収録。LLMはクエリへの回答時にまずこのファイルを読む
- コサイン類似度（ベクトル検索）は「似た単語」を見つけるが、LLMは「質問が本当に聞いていること」を理解して関連概念を選択できる
- 個人KBスケール（50〜500記事）ではインデックス全体がコンテキストウィンドウに収まるため、RAGのオーバーヘッドなしに全記事カタログを参照できる
- スケールの限界：約2,000記事・2Mトークンを超えるとインデックスがコンテキストウィンドウに収まらなくなる — その時点でハイブリッドRAGへの移行が必要
- SessionStart hookがインデックスと最新dailyログを自動でコンテキストに注入するため、毎セッション最新の知識状態が参照できる

## Details

インデックスベース検索の動作：クエリを受け取ったLLMはまず `knowledge/index.md` を読み込み、質問に関連する3〜10記事を特定する。次にそれらの記事を全文読み込んで回答を合成する。最後に `--file-back` オプションが指定されていれば、Q&A記事を `knowledge/qa/` に保存してインデックスとログを更新する。

RAGとの比較：RAG（Retrieval-Augmented Generation）はテキストをチャンクに分割しベクトル埋め込みにして類似度検索を行う。このアプローチは大規模DBでは有効だが、個人KBスケールでは設定コスト・誤検索・コンテキスト断片化などのデメリットが上回る。LLMによるインデックス読み取りはシンプルで、記事全体の文脈を保持したまま検索できる。

スケールアップ時の対応（将来）：~2,000記事を超えた時点でインデックスが大きくなりすぎる。その際はTobi Lutkeの `qmd` などのツールによるハイブリッド検索（キーワード＋セマンティック）をRAGとして追加するのが推奨される。
